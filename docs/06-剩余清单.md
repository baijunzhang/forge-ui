### A7. 所有滚动条拖动不丝滑
- 现象:文件列表、diff 面板、聊天区等多处滚动条,拖动时手感卡顿/跳动,不跟手。

  指令(发给 AI):
  ```text
  STEP P11 — Scrollbar dragging feels janky across the app; make it smooth.
  Frontend-only, investigate before fixing.

  INVESTIGATE first:
  1. Are the scrollbars in this app native browser scrollbars styled via CSS
     (::-webkit-scrollbar-thumb etc.), or a custom JS-driven scrollbar component that
     recalculates position on mousemove?
  2. If custom/JS-driven: check whether the thumb or track has any CSS `transition`
     property (e.g. `transition: all 0.2s`) applied — a transition on a property being
     updated in real time during drag is the single most common cause of laggy-feeling
     scrollbar drags, because each mousemove fights the transition's easing instead of
     moving instantly.
  3. If custom/JS-driven: check whether the drag handler updates scroll position
     directly on every `mousemove` event (can cause jank if not batched) versus using
     `requestAnimationFrame` to batch updates to the next paint frame.
  4. Identify EVERY place a custom scrollbar is used (file list, diff panel, chat feed,
     sidebar sessions list, etc.) — list them so a fix can be applied consistently
     everywhere, not just one.

  FIX (apply once diagnosed, likely some combination of):
  1. Remove any `transition`/`animation` on scrollbar thumb position, width, or
     transform during active dragging (transitions are fine on hover/color changes,
     not on position while the user is actively dragging).
  2. If drag position is computed on every mousemove without batching, wrap the
     position update in `requestAnimationFrame` so it's synced to the browser's paint
     cycle instead of firing faster than the screen can render.
  3. Make sure `user-select: none` is applied to the scrollbar thumb/track while
     dragging, so text selection doesn't fight with the drag gesture (a very common
     cause of "sticky"/jumpy scrollbar drags).
  4. If these are QWebEngineView-rendered scrollbars, also check whether hardware
     acceleration / compositing is enabled for the web view — software rendering can
     make even correct CSS/JS feel laggy. Report if this is configurable.

  Verify live: drag every scrollbar you found in step 4 (file list, diff panel, chat
  feed, sidebar) slowly and quickly, confirm the thumb tracks the cursor 1:1 with no
  lag or snapping, in both light and dark themes.
  ```

### B补充. Session 标题应该是"首次请求的摘要",而不是第一句话原文

  两个模式，一个是...")看起来就是第一条用户消息原文截断加省略号,不是真正总结过的标题,
  信息量不够,列表里不好一眼分辨每个 session 是干什么的。
- 目标:像 Claude/ChatGPT 那样,session 建立后自动生成一个**真正概括意图**的简短标题
  (比如 "Fix scrollbar drag jank" 而不是 "请把这个滚动条拖动的时候...")。这个不是
  纯前端能做的——需要额外调一次 LLM 做摘要,并把结果写回本地 session JSON 的 title
  字段,所以要先调查现在的存储/生成机制。

  指令(发给 AI):
  ```text

  and stored?
  1. Confirm: is the title today literally the first user message, truncated/ellipsized
     for display — or is there already some summarization step that just isn't very
     good? Report where this happens (file/line) and where the title is persisted
     (the local session JSON's "title" field, per earlier docs).
  2. Is there an existing lightweight/fast LLM call path already used elsewhere in this
     app (e.g. for something else small) that could be reused for a quick title-
     generation call, or would this need a new call path? Report options and rough
     cost/latency tradeoffs if you can estimate them.
  3. When would title generation naturally fire — right after the first user message is
     sent, or after the first assistant reply completes (using both messages usually
     produces a better summary than the user message alone)? Recommend one and why.
  Wait for my go-ahead before implementing.

  message verbatim.
  1. After the point recommended in P27 (first exchange completes is usually better
     than just the raw first message), make a short, low-cost LLM call whose ONLY job
     is producing a concise title: a few words, no trailing punctuation, capturing the
     intent of the request (e.g. "Fix scrollbar drag jank", "Multi-session concurrency
  ```
